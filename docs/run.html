The Joined Node task request endpoint has been set up to be flexible and to execute code in the Joined Node cluster:

hello.js 

HTTPS {verb} https://api.joinednode.com/run/{unique-container}/{task-id}

  1. The {unique-container} URL segment identifies the name of the container to execute the request in. This container is unique and linked only to your account.
  2. The {task-id} URL segment identifies the unique id of the task to execute. These tasks are added via the Joined Node Dashboard, and are associated with a single container.
  3. The {verb} must be POST, PATCH, PUT, or GET.

Now you Know...

The Joined Node task cluster may cache the task code obtained to reduce the latency of consecutive task requests.

If you do decide that you not want any caching (for example, when you frequently modify the code during development), you can specify the task_no_cache=1 URL query parameter when making the task request.

Using this parameter will force the task cluster to ignore any cached version of the task code and instead get the newest version.

HTTP Response Codes

Regardless of how you decide to run the task code, the HTTP response has contains either the result of code execution or an error, depending on the status code of the response as follows:

  • 200: Code executed correctly and response contains the result of the task running.
  • 400: There was a problem with the data submitted by the caller, the response body contains details.
  • 403: The task is not submitted with the request or does not grant sufficient permissions.
  • 429: Requests to the {unique-container} are throttled. The caller is expected to wait at least the amount of time specified in the Retry-After HTTP response header before re-attempting the request.
  • 500: Fatal execution error of the task, the response contains details.
