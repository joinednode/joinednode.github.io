The Joined Node recipe request endpoint has been set up to be flexible and to execute code in the Joined Node cluster:

hello.js 

HTTPS {verb} https://api.joinednode.com/run/{unique-container}/{recipe-id}

  1. The {unique-container} URL segment identifies the name of the container to execute the request in. This container is unique and linked only to your account.
  2. The {recipe-id} URL segment identifies the unique id of the recipe to execute. These recipes are added via the Joined Node Dashboard, and are associated with a single container.
  3. The {verb} must be POST, PATCH, PUT, or GET.

Now you Know...

The Joined Node recipe cluster may cache the recipe code obtained to reduce the latency of consecutive recipe requests.

If you do decide that you not want any caching (for example, when you frequently modify the code during development), you can specify the recipe_no_cache=1 URL query parameter when making the recipe request.

Using this parameter will force the recipe cluster to ignore any cached version of the recipe code and instead get the newest version.

HTTP Response Codes

Regardless of how you decide to run the recipe code, the HTTP response has contains either the result of code execution or an error, depending on the status code of the response as follows:

  • 200: Code executed correctly and response contains the result of the recipe running.
  • 400: There was a problem with the data submitted by the caller, the response body contains details.
  • 403: The recipe is not submitted with the request or does not grant sufficient permissions.
  • 429: Requests to the {unique-container} are throttled. The caller is expected to wait at least the amount of time specified in the Retry-After HTTP response header before re-attempting the request.
  • 500: Fatal execution error of the recipe, the response contains details.
